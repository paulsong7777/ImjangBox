# v0.2.0-alpha Release Notes

**Status:** published pre-release; validation passed
**Released:** 2026-06-12
**Previous published alpha:** `v0.1.0-alpha`
**Scope:** published alpha pre-release for completed Phase 0 through Phase 5 work

## Summary

`v0.2.0-alpha` packages the completed Phase 4 and Phase 5 work that landed after the `v0.1.0-alpha` tag. The pre-release keeps the product scope focused on broker-side commercial-property inspection capture, dynamic facility checks, map/geocoding/search boundaries, and privacy-safe customer share cards.

No Phase 6 product features are included.

## Highlights

- Customer share cards are generated from persisted public snapshot rows, not live internal inspection records.
- Public share pages render customer-safe address, pricing, verification, visible facility summaries, and share-scoped image references.
- Existing public share cards remain stable after internal inspection records change.
- Public image streaming uses `/share/{shareId}/images/{displayOrder}` and does not expose storage keys, local paths, or original filenames.
- Share-card generation writes internal audit rows with `CREATED` and `UPDATED` actions.
- MyBatis mapper XML has SQL-backed integration coverage through the `mybatis-integration` profile with H2 in MySQL compatibility mode.
- Broker upload validation enforces attachment count, size, allowed content type, filename extension/content-type consistency, and header/content-type consistency before storage.
- Operations documentation now covers profiles, required secrets, Kakao key separation, file/database backups, small-server deployment, smoke tests, and rollback.

## Privacy And Security

Public share output remains deny-by-default. The following internal fields must not appear in public share DTOs, templates, JSON responses, snapshots, image routes, or release examples:

- `private_memo`
- `price_private_note`
- `stakeholder.phone`
- `contact_log.content`
- `internal_risk_memo`

Additional public-share exclusions verified by the Phase 5 state include internal addresses, raw storage keys, local file paths, original filenames, contact-log content, and share audit rows.

Broker routes remain HTTP Basic protected and broker forms remain CSRF protected. TLS termination is required for any networked deployment because broker credentials currently use HTTP Basic.

## Deployment Notes

Use `docs/operations.md` as the operational source of truth.

- Default `local` profile uses in-memory mappers and local file storage.
- Production-like small-server runs should use `--spring.profiles.active=local-db` with MySQL, Flyway, and environment-supplied secrets.
- There is no dedicated `prod` profile yet.
- Keep `KAKAO_REST_API_KEY` server-side and separate from `KAKAO_MAP_JAVASCRIPT_KEY`.
- Back up MySQL and the configured file-storage root together before deployment.
